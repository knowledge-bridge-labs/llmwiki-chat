# Support

Use public GitHub issues for reproducible product bugs, focused feature
requests, documentation gaps, and adapter compatibility reports.

Before opening an issue:

1. Check the README and adapter documentation for the expected runtime and
   Knowledge Source endpoint contract.
2. Try the smallest public reproduction that does not include credentials,
   private endpoint URLs, raw sensitive logs, customer content, or screenshots
   exposing private infrastructure.
3. Use the issue form that matches the request.

For suspected vulnerabilities, do not open a public issue. Follow
[SECURITY.md](SECURITY.md).

For answer-quality reports, include whether the result came from `Custom A2A` or
a named external runtime, plus the non-sensitive Knowledge Source protocol and
runtime setup used. `Mock Agent` reports should focus on UI, trace, citation, or
graph behavior; mock answer text is not treated as a production answer-quality
baseline.
